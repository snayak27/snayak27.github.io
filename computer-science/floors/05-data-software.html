Floor 5 · the craft

Data, Algorithms & Software

This is the floor most people mean when they say "coding." It is where ideas become working systems: organising data so it can be found instantly, choosing algorithms that stay fast at scale, and writing the software that runs everything you use.

🧱 What this floor is

Floor 5 sits between the machine below and the people above. It covers three crafts: data structures (how information is arranged), algorithms (the step-by-step methods that act on it), and software engineering (building large systems that don't collapse under their own weight). It is where the abstract rules of Floor 1 turn into apps, databases and tools.

The big ideas

How we structure information, query databases, and write human-readable code.

⚡ 1. Algorithms & the cost of doing things (Big-O)

An algorithm is a precise recipe for solving a problem, like a guide to baking cookies. The crucial question is: "Does it still work when the data is huge?" Engineers measure this growth using Big-O notation.

To store data, we choose different **Data Structures** based on how we need to access them:

  • 🗄️ Arrays (Numbered lockers): Storing items in a contiguous line. It is instant ($O(1)$) to open locker #5. But if you want to insert a new locker in the middle, you have to slide all the lockers after it down, which is slow ($O(N)$).
  • 🔗 Linked Lists (Treasure hunt): Storing items where each item has a pointer to the next one. It is very easy to insert a new clue in the middle ($O(1)$), but to find a clue at the end, you have to follow the chain from the very beginning ($O(N)$).
  • 🔑 Hash Tables (Indexed mailroom): Using a hash function to map keys directly to specific slots. This gives near-instant ($O(1)$) average speeds for searching, inserting, and deleting.

🔎 Where you meet it

Finding a contact by jumping to the right part of an alphabetised list is near-instant even with a million names; scanning every name one by one would crawl. Choosing the first approach over the second is exactly what algorithm design is about.

🔗 How it connects to other floors

These data structures are blueprints for organizing physical memory in RAM (Floor 3) and on hard drives (Floor 2). Choosing the wrong structure can slow down a network server (Floor 4) or cause an app (Floor 7) to freeze.

🗄️ 2. Storing data so it survives and scales (databases)

In 1970, Edgar CoddIBM researcher whose 1970 paper introduced the relational model — organising data into linked tables. It underpins SQL and almost every business database since. at IBM proposed the relational model.

Think of it as a **linked family of spreadsheets**. Instead of repeating a student's address, parent names, and phone numbers every single time they borrow a library book, you keep a "Students" sheet and a "Library Books" sheet, and draw a line (a relational key) between them. This keeps the data clean and avoids repeating the same information.

To avoid scanning every row of data one-by-one, databases build **B-Tree Indexes**. Imagine a B-Tree as a **physical filing cabinet** with labels: the top drawer is labeled A-M and N-Z. If you look for "Sam," you skip the first drawer completely, open the second, and check alphabetical tabs inside. Instead of opening 10,000 files, you find Sam in 3 steps!

To keep data safe, database systems guarantee ACID rules:

  • 💥 Atomicity (All-or-Nothing): If you transfer money, it either leaves your account and arrives in your friend's account, or the whole transaction is cancelled. It never gets lost in transit.
  • 📜 Consistency: The database must obey rules (like: an account balance cannot go below zero).
  • 🔒 Isolation: If two people buy the same seat at the same time, the database forces one to go first, preventing double booking.
  • 💾 Durability: Once a transaction is saved, it is written to the physical disk. Even if the power cuts out a second later, the data is permanent.
🔎 Where you meet it

When a payment either completes fully or not at all — never leaving money half-moved — that guarantee comes from database rules first formalised on this floor in 1970.

🔗 How it connects to other floors

Databases translate high-level SQL queries (Floor 5) into raw reads and writes handled by the operating system kernel (Floor 3) on physical storage drives (Floor 2), and sync copies of data across servers worldwide (Floor 4).

The real need

Earlier databases hard-wired the data to its storage layout on disk. Change how it was stored and every program that used it broke.

What it forced

Codd's relational model gave data independence — ask questions of the data in simple tables without knowing how it's physically stored. It became SQL, and the backbone of business computing.

🛠️ 3. Building systems that last (software engineering)

A small script is easy; a system with millions of lines, hundreds of contributors and millions of users is a different challenge entirely. Software engineering is the discipline of structuring code, testing it automatically, and changing it safely over years.

To convert human code into something hardware understands, we use:

  • 📕 Compilers: Translate the entire codebase at once into binary machine instructions (like Rust, C, or C++). The result runs incredibly fast, but you must compile again every time you make a change.
  • 🗣️ Interpreters: Translate code line-by-line on the fly as it runs (like Python or JavaScript). It is slower to execute, but lets you quickly test changes as you type.
⚡ The 2017 connection — "Software 2.0"

Traditionally, coding is **Software 1.0**: a programmer writes a detailed, 500-page rule manual telling a robot how to find a red ball.

After 2017, **Software 2.0** took over: instead of writing the rules, we show the robot 10,000 photos of red balls and let it **train a neural network** to find the rules itself. The program is *learned* from data rather than coded line-by-line.

🎮 Try it yourself: The Handshake Count

Let's compare how two algorithms sort a hand of **5 playing cards** (numbers: [4, 1, 5, 2, 3]):

  1. 🃏 Selection Sort (Simple but slow): Look at all 5 cards, find the smallest (1), swap it to the front. Now look at the remaining 4, find the next smallest, swap it. Total checks: 4 + 3 + 2 + 1 = 10 checks. If you had 1,000 cards, this would take about **500,000 checks**! (O(N²))
  2. 🥞 Merge Sort (Clever and fast): Split the 5 cards into individual piles. Merge adjacent piles together in order. For 1,000 cards, this takes only about **10,000 checks**! (O(N log N))
🔗 How it connects to other floors

Human-readable languages (Floor 5) are compiled into machine instruction sets (Floor 3) to run on CPU silicon (Floor 2). In Software 2.0, neural networks (Floor 6) write some of the code for user interfaces (Floor 7).

How this floor evolved

From hand-coded logic and file-storage to data-driven Software 2.0.

Then

Hand-written rules

Every behaviour was spelled out explicitly in code. Data lived in rigid files, and large projects were built mostly from scratch.

Now

Reusable parts & huge data

Software is assembled from libraries and services; databases hold oceans of data; automated testing keeps massive systems stable.

Next

Trained, not coded

For many tasks, models trained on data replace hand-written rules — and AI assistants now help write the remaining code.

1952
The First Compiler (A-0) ↗

Grace Hopper designs the A-0 compiler, letting programmers write symbolic instructions that are automatically translated to binary machine numbers.
1957
FORTRAN — high-level language ↗

John Backus and IBM release FORTRAN (Formula Translation), letting engineers write mathematical equations instead of raw machine numbers.
1958
LISP is created ↗

John McCarthy invents LISP, introducing functional programming and trash collection (garbage collection), which became the language of AI research.
1970
Codd's relational model ↗

Edgar Codd proposes organizing database information into linked tables — the idea behind SQL and nearly every enterprise database since.
1974
SQL is designed ↗

Donald Chamberlin and Raymond Boyce design Structured Query Language (SQL) at IBM to access and manipulate data stored in relational databases.
1991
Python is released ↗

Guido van Rossum releases Python, prioritizing code readability and simplicity, making it the dominant scripting and AI language.
2005
Git version control ↗

Linus Torvalds designs Git to manage Linux kernel development, creating the standard distributed version control system for software teams.
2010
Rust Language ↗

Graydon Hoare and Mozilla introduce Rust, prioritizing memory safety and concurrent execution without a runtime garbage collector.
2013
Docker containerization ↗

Solomon Hykes and Dotcloud release Docker, standardizing how server programs are packaged with their dependencies into isolated containers.
2021
GitHub Copilot — AI-assisted coding ↗

GitHub launches Copilot, an AI code assistant that suggests code inline, marking the beginning of the Software 2.0 shift in development.

⚡ This floor's role in the 2017 cascade

Floor 5 is where the cascade changed the job itself. The most powerful way to build certain software flipped from "write the rules" to "assemble the data and train the model." Data — once just something software acted on — became the raw material that the program is made from. The craft of software is being rewritten from the AI floor down.
